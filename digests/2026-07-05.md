## Engineering & AI Workflows

- [sqlite-utils 4.0rc2, mostly written by Claude Fable (for about $149.25)](https://simonwillison.net/2026/Jul/5/sqlite-utils-fable/#atom-everything) — Simon Willison used Claude Fable via Claude Code for a pre-release review and it caught a severe data-loss bug (`delete_where()` poisoning the connection and silently dropping all subsequent writes) that had been missed entirely; 37 prompts, 34 commits, 30 files changed — a practical case study in using an agent for final QA rather than greenfield generation.

- [Better Models: Worse Tools](https://simonwillison.net/2026/Jul/4/better-models-worse-tools/#atom-everything) — Opus 4.8 and Sonnet 5 are *more likely* than older Claude models to hallucinate extra fields in tool call schemas when using third-party coding harnesses; the likely cause is RL training specifically targeting Claude Code's own edit tool schema, making those models a worse fit for custom tool interfaces — a concrete AI code quality regression to watch for if you're building or using non-Claude-Code agents.

- [Mapping with In-Memory Layers to Reduce LLM Overload](https://ridgetext.com/blog/mapbox-llm-composition) — A practical pattern for tool design: never return large data payloads through an LLM; instead, store results server-side and return only lightweight acknowledgments (layer IDs, status, counts), letting the LLM orchestrate by reference rather than by value. Directly applicable to any agent that touches databases, files, or APIs returning bulk data.

- [Using "applications" to make a smaller model more effective at bigger tasks](https://www.reddit.com/r/LocalLLaMA/comments/1unobl4/using_applications_to_make_a_smaller_model_more/) — Scoping agents into constrained "application" views with a limited, menu-driven toolset (e.g. "open 1, copy 2") rather than exposing 100+ tools at once reduced errors on exact-text tasks (URLs, identifiers) and surprisingly made a 4B model outperform a 26B on the same workflow — a useful framing for prompt/tool architecture.

## Try This
Before your next major release or PR merge, open a Claude Code session and prompt: *"Act as a senior code reviewer doing a final pre-release audit. Focus specifically on: (1) any method that writes to a database or file but lacks explicit transaction/commit handling, (2) any breaking changes that aren't reflected in the version bump, and (3) edge cases in error paths that could silently discard data. List findings as release blockers, significant issues, or minor notes."* Simon Willison used almost exactly this pattern with Fable and caught a connection-poisoning data-loss bug that had survived all prior review — the cost was $149 and the return was avoiding a 5.0 forced version bump.
