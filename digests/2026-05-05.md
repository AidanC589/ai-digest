## General AI Developments
- [Model Organisms Are Leaky: Perplexity Differencing Often Reveals Finetuning Objectives](https://arxiv.org/abs/2605.00994) — Researchers show a simple perplexity-gap method can expose what a model was finetuned to do (including hidden or unsafe behaviors) without access to model internals, tested across 76 model organisms up to 70B parameters; works even with a substitute reference model, raising real concerns about detectability of covert fine-tunes in AI-generated code pipelines.

- [FastDMS: 6.4X KV-cache compression running faster than vLLM BF16/FP8](https://www.reddit.com/r/LocalLLaMA/comments/1t3vlrx/fastdms_64x_kvcache_compression_running_faster/) — MIT-licensed FastDMS achieves 6x+ KV-cache compression with 1.5–2x decode speedups over vLLM baselines at near-lossless quality; practically relevant if you're running local inference for coding assistants.

## Engineering & AI Workflows
- [TRE Python binding — ReDoS robustness demo](https://simonwillison.net/2026/May/4/tre-python-binding/#atom-everything) — Simon Willison had Claude Code build a ctypes Python binding to the TRE regex library and then attack it with malicious patterns; TRE handled "evil" regexes on 10M-character inputs faster than Python's `re` on tiny ones — a useful template for using AI assistants to both build *and* adversarially test security-sensitive code in one session.

- [[AINews] The Other vs The Utility](https://www.latent.space/p/ainews-the-other-vs-the-utility) — A nuanced industry discussion on Claude's "character" versus GPT's tool-like utility; the practical observation is that users route different query types to different models based on perceived judgment — worth considering when designing multi-model workflows and which assistant you put in which review loop.

- [CLEAR: Revealing How Noise and Ambiguity Degrade Reliability in LLMs for Medicine](https://arxiv.org/abs/2605.01011) — Across 17 LLMs, adding more plausible answer options or softer abstention framing ("I don't know") consistently degrades accuracy and increases wrong confident answers; the "humility deficit" worsens at scale. Direct implication: when prompting AI for code review or security analysis, avoid open-ended option lists and be explicit about abstention conditions.

- [Can AI Debias the News? LLM Interventions Improve Cross-Partisan Receptivity but LLMs Overestimate Their Own Effectiveness](https://arxiv.org/abs/2605.01006) — LLMs significantly overestimate how effective their own outputs are compared to measured human responses, and their internal model of *who* responds to interventions is wrong. Relevant for anyone using AI self-evaluation or self-critique loops in code review — the model's confidence in its own review quality is likely inflated.

## Try This
Use Claude Code to both *build* and *attack* a component in the same session: open a session, ask it to implement something security-sensitive (input parsing, regex handling, auth logic), then immediately follow up with "Now write a test harness that tries to break this with malicious/adversarial inputs and report any failures." Willison's ReDoS demo shows this works well — the model has full context of what it just built, so its attack surface knowledge is accurate rather than generic. End the session by asking it to summarize any vulnerabilities it found and suggest mitigations before you review the diff.
