## Engineering & AI Workflows
- [[AINews] The Field Guide to Fable](https://www.latent.space/p/ainews-the-field-guide-to-fable) — Practical breakdown of working with Fable-class Claude models: "unhobbling" means auditing and removing outdated prompt constraints that suppress new capabilities; specific techniques include a `blindspot pass` instruction, maintaining a running `implementation-notes.md` of decisions made autonomously by the agent, and using `quiz me` to verify your own understanding of AI-generated code — all immediately applicable to Claude Code sessions.
- [Qwen 3.6 27B absolutely fails at agentic work](https://www.reddit.com/r/LocalLLaMA/comments/1uphzhj/qwen_36_27b_absolutely_fails_at_agentic_work/) — A practitioner running Qwen 3.6 27B at 8-bit and 16-bit reports consistent agentic failure every ~4 turns despite strong single-prompt performance; reverted to 122B for reliable multi-step tasks. A useful reminder that benchmark scores on isolated prompts don't predict multi-turn agent reliability — always test locally on your actual workflow length before switching models.
- [Improving LLMs via Validator-to-Generator Alignment](https://arxiv.org/abs/2607.02668) — New research on the generator-validator gap shows LLMs frequently generate outputs they'd flag as invalid if asked to review them separately — up to +27pp improvement in consistency achievable via training. Practical implication: when reviewing AI-generated code, explicitly asking the model to validate its own output in a *separate prompt* (not the same turn) may surface more real issues than inline self-critique.
- [OfficeCLI: Office suite for AI agents to read and edit Microsoft Office files](https://github.com/iOfficeAI/OfficeCLI) — CLI tool giving AI agents structured read/write access to .docx, .xlsx, and .pptx files; relevant if your Claude Code workflows need to ingest specs, requirements docs, or generate reports without converting to markdown first.

## Tool Updates
- [Claude Code v2.1.202](https://github.com/anthropics/claude-code/releases/tag/v2.1.202) — Notable changes: new `/config` setting for **Dynamic workflow size** (small/medium/large agent counts) gives you explicit control over how aggressively Claude spawns sub-agents; `/review <pr>` is reverted back to fast single-pass — use `/code-review <level> <pr#>` if you want the multi-agent deep review. ## Try This
In your next Claude Code session, add a `implementation-notes.md` file to your repo and instruct Claude to append a one-line entry every time it makes an underspecified design decision autonomously — something like: *"Whenever you make an architectural or implementation choice that I haven't explicitly specified, append a timestamped entry to `implementation-notes.md` describing what you decided and why, before proceeding."* Review this file at the end of each session instead of trying to reconstruct decisions from the conversation. This directly addresses the "invisible drift" problem in AI-assisted codebases where the agent's judgment calls accumulate silently and become hard to audit or reverse.
