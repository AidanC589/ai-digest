## Engineering & AI Workflows

- [**Rewriting Bun in Rust**](https://simonwillison.net/2026/Jul/8/rewriting-bun-in-rust/#atom-everything) — Jarred Sumner's account of rewriting Bun's entire Zig codebase into Rust using Claude agents in 11 days is essential reading: the key enabler was a language-independent TypeScript test suite acting as a conformance harness, combined with adversarial code review and fixing the *code generation process* rather than hand-patching individual outputs. Cost: ~$165K at API pricing, 5.9B uncached input tokens. The result has been live in Claude Code since June 17th — startup 10% faster on Linux, and "boring is good."

- [**Quoting Kenton Varda**](https://simonwillison.net/2026/Jul/8/kenton-varda/#atom-everything) — Cloudflare's Kenton Varda banned AI-written PR/commit descriptions from his team because they describe *what* the code does (already visible in the diff) while omitting *why* — the higher-level framing actually needed for review. A useful forcing function: treat AI-generated change descriptions as a review anti-pattern, not a time-saver.

- [**Why AI Infrastructure must evolve for Agent Experience — Akshat Bubna, Modal CTO**](https://www.latent.space/p/modal2026) — Modal's CTO argues that traditional cloud infra (built for humans who can read docs and reason through errors) breaks down for agents, which need fast iteration loops, sandboxes, and hard guardrails. The key insight for workflow design: observability matters more than reading code when agents are the ones writing it, and RL rollouts can require 100,000 concurrent sandboxes.

- [**GLM-5.2 fearmongering in the press**](https://www.reddit.com/r/LocalLLaMA/comments/1urhzox/glm52_fearmongering_in_the_press/) — Security firms Semgrep and Graphistry found GLM-5.2 (a fully open-weight model) proficient at identifying software bugs and cybersecurity tasks at near-Mythos levels — relevant if you're thinking about supply chain risks from open-weight models being used in adversarial code review contexts.

## Try This

The Bun rewrite makes explicit what makes agentic code generation trustworthy at scale: you fix the *process*, not the *output*. Apply this to your own Claude Code workflows by keeping a `AGENT_ISSUES.md` in your repo — whenever you catch a recurring pattern of bad output (wrong abstraction level, missed error paths, incorrect typing), write a new standing instruction in your `CLAUDE.md` or system prompt rather than fixing the individual file. Then verify the fix holds on the next generation run. This shifts your review energy from patching symptoms to improving the generation loop, which compounds over time rather than resetting every session.
