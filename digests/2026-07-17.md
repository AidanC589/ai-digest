The author also uses a consensus pattern — asking Claude Opus, Fable, and Codex to independently review configs and flag disagreements — which is a transferable review technique for any AI-generated infrastructure code.

- **[Just Keep Prompting: Evaluating Repetitive Socratic Prompting in VLMs](https://arxiv.org/abs/2607.14099)** — Peer-reviewed finding: repeatedly challenging a model's answer (Socratic interrogation) is more likely to destabilize a correct answer than to fix a wrong one. Across 720 multi-turn runs, models flip answers repeatedly rather than converging. Implication for code review: don't keep re-prompting when you disagree with AI output — it produces confident wrongness, not better reasoning.

## Tool Updates

- **[Claude Code v2.1.212](https://github.com/anthropics/claude-code/releases/tag/v2.1.212)** — Substantial release. Key changes: `/fork` now creates a background session (not an in-session subagent); new per-session caps on WebSearch (default 200) and subagent spawns (default 200) to stop runaway loops; MCP tool calls >2 minutes auto-background; `/resume` now shows a full picker of past sessions including deleted ones. Security fixes: plan mode was silently auto-running file-modifying bash commands (`touch`, `rm`) without permission prompts — now fixed. Also fixes SIGTERM orphaning processes and `/ultrareview` failing on PR references like `#123`.

- **[Codex 0.145.0-alpha.18–.20](https://github.com/openai/codex/releases/tag/rust-v0.145.0-alpha.20)** — Three rapid alpha releases this week with no release notes published. Given the active `$HOME` deletion bug in GPT-5.6 + full-access mode, treat these as unstable and keep sandboxing enabled until a stable release with explicit bug-fix notes appears.

## Try This

Cross-model config consensus review: when Claude Code generates infrastructure or configuration code (Terraform, Docker, MikroTik, CI pipelines), paste the output into two other frontier models — e.g., Kimi K3 via API and GPT-5.6 Sol — with the prompt: *"Review this config for correctness, security issues, and anything missing. Flag any line you'd change and why."* Only accept changes that two of three models independently flag. This exploits model diversity to surface blind spots without the "Socratic destabilization" failure mode (where re-prompting the same model just produces confident flip-flopping). The MikroTik networking post validated this pattern in production; the JKP research explains why it works better than iterating on one model alone.
